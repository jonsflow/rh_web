Component Relationships

SmartDataFetcher → uses → OptionsDatabase (composition)
OptionService → uses → DataRepository (composition)
OptionService → uses → PnLCalculator (composition)
OptionService → uses → PositionClassifier (composition)
OptionService → creates → Position objects (factory pattern)
DataRepository → queries → SQLite Database (data access)
OptionsDatabase → delegates to → OptionService (facade pattern)

Flask API Endpoints

GET /api/options → SmartDataFetcher.get_processed_data()
POST /api/update → SmartDataFetcher.update_data()
GET /api/daily-pnl → OptionService.get_daily_pnl_summary()
GET /api/positions/date/<date> → OptionService.get_positions_by_date()

Typical Data Processing Flow

1. rh_web.py receives API request
2. SmartDataFetcher coordinates the operation
3. OptionService retrieves raw orders via DataRepository
4. PnLCalculator computes position P&L
5. PositionClassifier determines position status
6. Position objects are created and returned
7. Response serialized and sent to frontend
Key Architecture Patterns:
• Service Layer: Business logic separated from data access
• Repository Pattern: DataRepository abstracts database operations
• Factory Pattern: OptionService creates Position objects
• Static Utility Classes: PnLCalculator, PositionClassifier
• Data Models: Position, OptionOrder encapsulate data structure
• Facade Pattern: OptionsDatabase provides simplified interface
